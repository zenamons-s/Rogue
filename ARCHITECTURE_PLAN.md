# Архитектурный план портирования Rogue-like игры на Go

## Введение
На основе анализа примеров кода на C (`code-samples/rogue_sample` и `code-samples/task_samples`) разработан план портирования игры Rogue (1980) на язык Go с соблюдением принципов чистой архитектуры и многослойного разделения.

## Цели
- Создать консольную игру с управлением с клавиатуры.
- Реализовать core-механики классического Rogue.
- Обеспечить чёткое разделение на слои: Domain, Presentation, Data.
- Поддержать расширяемость для bonus-заданий.

## Анализ примеров
Примеры на C демонстрируют:
- **Многослойную архитектуру** с разделением на domain, presentation, data.
- **Процедурную генерацию** уровней (комнаты, коридоры).
- **Развитую систему сущностей**: игрок, враги, предметы.
- **Рендеринг через ncurses** с поддержкой 2D и 3D (ray casting).
- **Сохранение состояния** в JSON.

## Предлагаемая архитектура на Go

### Слои
1. **Domain Layer** (бизнес-логика)
   - Пакет `entities`: структуры данных.
   - Пакет `generation`: генерация уровней.
   - Пакет `gameplay`: механики игры (движение, бой, инвентарь).
   - Пакет `services`: сервисы для координации логики.

2. **Presentation Layer** (пользовательский интерфейс)
   - Пакет `render`: отрисовка через библиотеку `tcell`.
   - Пакет `ui`: компоненты интерфейса (меню, HUD).
   - Пакет `input`: обработка ввода с клавиатуры.

3. **Data Layer** (работа с данными)
   - Пакет `storage`: репозитории для сохранения/загрузки игры, статистики.
   - Формат хранения: JSON.

4. **Infrastructure** (внешние зависимости)
   - Библиотека `github.com/gdamore/tcell/v2` для терминального UI.
   - Стандартные библиотеки Go для JSON, случайных чисел, времени.

## Зависимости
- **Go 1.21** или выше.
- **tcell v2** для терминального интерфейса.
- **testify** для тестирования (опционально).

Файл `go.mod`:
```go
module rogue-game

go 1.21

require (
    github.com/gdamore/tcell/v2 v2.7.0
    github.com/stretchr/testify v1.8.4 // for testing
)
```

## План реализации
Детальный план разбит на 9 этапов (см. `implementation_plan.md`):

1. **Подготовка окружения** – настройка проекта, структуры каталогов.
2. **Domain Layer: Сущности** – определение структур данных.
3. **Domain Layer: Генерация уровней** – алгоритмы процедурной генерации.
4. **Domain Layer: Игровая механика** – движение, бой, предметы.
5. **Data Layer: Сохранение/загрузка** – JSON-сериализация.
6. **Presentation Layer: Рендеринг** – отрисовка карты, интерфейса.
7. **Интеграция** – сборка игрового цикла, меню.
8. **Bonus-задания** – двери, динамическая сложность, мимик, 3D.
9. **Тестирование и документация** – unit-тесты, финализация.

Ориентировочный срок: 8 недель.
